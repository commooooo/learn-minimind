## Q1: 混合精度训练的原理？为什么不全用 FP16/BF16？

**标准答案：**

混合精度训练（Mixed Precision Training）的核心思想是：**前向传播和反向传播使用低精度（FP16/BF16），主权重（Master Weights）保持 FP32**。

具体流程：
1. 维护一份 FP32 的主权重副本（Master Weights）
2. 每次训练迭代时，将权重转换为 FP16/BF16 进行前向和反向传播
3. 用 FP16/BF16 计算出的梯度更新 FP32 主权重
4. 重复以上过程

为什么不能全用 FP16/BF16？
- **梯度下溢**：FP16 的最小正数约为 $6 \times 10^{-8}$，很多梯度值会被截断为 0，导致参数无法更新
- **权重更新消失**：当 `weight += lr * gradient` 中 `lr * gradient` 远小于 `weight` 的精度时，加法结果不变（精度不够表达差值）
- **累积误差**：长时间训练后低精度的舍入误差会不断累积，导致模型质量下降

混合精度的收益：
- 显存减少约 40-50%（激活值和中间结果用半精度存储）
- 计算速度提升 2-3x（现代 GPU 的 Tensor Core 对半精度有硬件加速）
- 训练精度基本不受影响（关键操作仍用 FP32）

**MiniMind 中的实现：**

MiniMind 使用 PyTorch AMP 实现混合精度训练，前向传播在 `autocast` 上下文中以 `bfloat16` 进行：

```python
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()

for batch in dataloader:
    with autocast(dtype=torch.bfloat16):
        logits = model(input_ids)
        loss = compute_loss(logits, labels)

    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()
    optimizer.zero_grad()
```

优化器内部自动维护 FP32 主权重，`autocast` 只将参与矩阵乘法的权重临时转换为 BF16 用于计算。

**追问方向：**
- 哪些操作适合在低精度下运行？哪些必须保持 FP32？（矩阵乘法适合低精度；softmax、layer norm、loss 计算需要 FP32）
- 混合精度训练比纯 FP32 的显存节省是如何计算的？

## Q2: FP16 vs BF16 的区别？为什么 LLM 训练更倾向 BF16？

**标准答案：**

| 特性 | FP16 | BF16 |
|------|------|------|
| 符号位 | 1 位 | 1 位 |
| 指数位 | 5 位 | 8 位 |
| 尾数位 | 10 位 | 7 位 |
| 数值范围 | $\pm 6.5 \times 10^4$ | $\pm 3.4 \times 10^{38}$（与 FP32 相同） |
| 精度（尾数） | 较高 | 较低 |
| 是否需要 Loss Scaling | 通常需要 | 通常不需要 |

LLM 训练更倾向 BF16 的核心原因：

1. **数值范围与 FP32 一致**：BF16 拥有 8 位指数，表示范围与 FP32 相同，几乎不会出现 overflow/underflow，训练更稳定
2. **不需要 Loss Scaling**：FP16 的窄范围在 LLM 这种深层网络中容易出现梯度下溢，必须搭配 GradScaler；BF16 通常无此问题
3. **LLM 对精度不敏感**：LLM 参数量巨大，单个参数精度略低（7 位 vs 10 位尾数）的影响可被统计平均效应消弭
4. **硬件支持**：A100、H100、3090 等训练卡都原生支持 BF16 Tensor Core

**MiniMind 中的实现：**

MiniMind 选择 `torch.bfloat16` 作为训练精度：

```python
with torch.autocast(device_type=args.device, dtype=torch.bfloat16):
    output = model(input_ids, labels=labels)
    loss = output.loss
```

同时仍保留了 `GradScaler`，这是一种防御性编程风格——BF16 下 scaler 实际上几乎不会触发缩放（因为不太可能出现 inf/nan），但保留它可以在切换回 FP16 时无缝支持。

**追问方向：**
- 如果 GPU 不支持 BF16（如 V100），应该如何处理？（用 FP16 + GradScaler）
- BF16 精度更低，为什么最终模型质量和 FP32 差不多？（梯度的统计性质 + 大量参数的平均效应）

## Q3: GradScaler 的作用和原理？什么时候需要它？

**标准答案：**

GradScaler（梯度缩放器）解决的核心问题是 **FP16 下的梯度下溢（underflow）**。

原理：
1. **前向缩放**：将 loss 乘以一个较大的 scale factor（初始通常为 $2^{16}$），使得反向传播中的梯度值被等比例放大，避免小梯度被截断为 0
2. **反向缩放**：在优化器更新参数前，将梯度除以 scale factor 还原到真实值
3. **动态调整**：每隔若干步（默认 2000 步）检查梯度中是否有 inf/nan
   - 无 inf/nan → 尝试增大 scale factor（乘以 `growth_factor`，通常 2.0）
   - 有 inf/nan → 跳过本次更新，减小 scale factor（乘以 `backoff_factor`，通常 0.5）

什么时候需要：
- 使用 FP16 训练时 **必须使用** GradScaler
- 使用 BF16 训练时 **通常不需要**（BF16 范围与 FP32 相同，不易下溢）
- 使用纯 FP32 训练时 **不需要**

**MiniMind 中的实现：**

MiniMind 使用 `scaler.scale(loss).backward()` 进行缩放反向传播，`scaler.unscale_()` 在梯度裁剪前还原梯度：

```python
scaler = torch.amp.GradScaler()

# 反向传播时 loss 被 scale 放大
scaler.scale(loss).backward()

# 裁剪前必须先 unscale，否则裁剪阈值不匹配
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# 更新参数（内部会再次检查 inf/nan）
scaler.step(optimizer)
scaler.update()
```

注意 `scaler.unscale_()` 的调用顺序——必须在 `clip_grad_norm_` 之前，否则梯度仍处于缩放状态，裁剪阈值就不是预期的 `max_norm=1.0` 了。

**追问方向：**
- 如果忘记调用 `scaler.unscale_()` 直接做 `clip_grad_norm_`，会发生什么？（梯度被 scale factor 放大，裁剪阈值实际变成了 1.0/scale_factor，训练几乎不更新）
- GradScaler 检测到 inf/nan 时跳过更新，这对训练有什么影响？（极少量 skip 无影响，频繁 skip 需排查问题）

## Q4: AMP（Automatic Mixed Precision）的使用方法和注意事项？

**标准答案：**

AMP 是 PyTorch 提供的自动混合精度训练机制，主要由两个组件构成：

1. **`torch.autocast`（或 `torch.cuda.amp.autocast`）**：
   - 自动将符合条件的操作转为低精度
   - 矩阵乘法（`mm`、`bmm`、`linear`）→ 转为 FP16/BF16
   - 规约操作（`softmax`、`layer_norm`、`cross_entropy`）→ 保持 FP32
   - 逐元素操作 → 跟随输入精度

2. **`torch.amp.GradScaler`**：
   - 配合 FP16 防止梯度下溢
   - BF16 下可选

使用注意事项：
- `autocast` 只包裹前向传播和 loss 计算，**不要包裹反向传播**
- 不要在 `autocast` 上下文中手动执行 `.half()` 或 `.float()` 转换
- Loss 必须在 `autocast` 内部计算
- 参数更新（`optimizer.step()`）必须在 `autocast` 外部
- 自定义 CUDA kernel 需要手动处理精度

**MiniMind 中的实现：**

MiniMind 的完整 AMP 使用模式：

```python
scaler = GradScaler()

for step, (input_ids, labels) in enumerate(loader):
    input_ids = input_ids.to(args.device)
    labels = labels.to(args.device)

    # autocast 只包裹前向传播 + loss 计算
    with autocast_ctx:
        res = model(input_ids, labels=labels)
        loss = res.loss + res.aux_loss
        loss = loss / args.accumulation_steps

    # 反向传播在 autocast 外部
    scaler.scale(loss).backward()

    if step % args.accumulation_steps == 0:
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        scaler.step(optimizer)
        scaler.update()
        optimizer.zero_grad()
```

MiniMind 的 `res.aux_loss` 来自 MoE（Mixture of Experts）的负载均衡损失，它与主 loss 一起在 `autocast` 内计算，保证梯度计算图完整。

**追问方向：**
- `autocast` 是线程安全的吗？多线程 DataLoader 下有没有问题？（`autocast` 是 per-thread 的，DataLoader 的 workers 是独立进程，不受影响）
- 如何验证 AMP 确实在用低精度计算？（检查中间张量的 dtype，或用 `torch.autocast` 的 `enabled` 参数做对比实验）

## Q5: 梯度累积的原理和实现？等效 batch_size 如何计算？

**标准答案：**

梯度累积（Gradient Accumulation）用于在显存受限时模拟大 batch 训练。

原理：不在每个 mini-batch 后立即更新参数，而是累积多个 mini-batch 的梯度后再统一更新。由于 `.backward()` 会将梯度**累加**到 `.grad` 上（默认不清零），多次反向传播的效果等价于对大 batch 求梯度。

等效 batch_size 计算：

$$\text{effective\_batch\_size} = \text{batch\_size} \times \text{accumulation\_steps} \times \text{num\_gpus}$$

例如 MiniMind 预训练：`batch_size=32, accumulation_steps=8, 1 GPU` → 等效 batch_size = 256。

关键细节：loss 必须除以 `accumulation_steps`。因为 PyTorch 的 `cross_entropy` 默认对 batch 内取平均（`reduction='mean'`），梯度累积是把多个 mini-batch 的平均 loss 求和。为了得到大 batch 的平均 loss，需要额外除以累积步数。

**MiniMind 中的实现：**

```python
for step, (input_ids, labels) in enumerate(loader):
    with autocast_ctx:
        res = model(input_ids, labels=labels)
        loss = res.loss + res.aux_loss
        loss = loss / args.accumulation_steps  # 关键：除以累积步数

    scaler.scale(loss).backward()  # 梯度累加到 .grad 上

    if step % args.accumulation_steps == 0:  # 累积够了再更新
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        scaler.step(optimizer)
        scaler.update()
        optimizer.zero_grad()  # 更新后清零
```

注意 MiniMind 的实现中，`optimizer.zero_grad()` 在更新后调用，确保下一轮累积从零开始。

**追问方向：**
- 梯度累积和大 batch 在数学上完全等价吗？（不完全等价——BatchNorm 的统计量不同；但 LLM 通常用 RMSNorm，没有此问题）
- 如果忘记 `loss / accumulation_steps`，会发生什么？（等效学习率变大 `accumulation_steps` 倍，训练可能不稳定）

## Q6: 梯度裁剪（Gradient Clipping）的原理？为什么用 max_norm=1.0？

**标准答案：**

梯度裁剪用于防止**梯度爆炸**。最常用的方式是**按范数裁剪**（Clip by Global Norm）：

1. 计算所有参数梯度的全局 L2 范数：$||\mathbf{g}||_2 = \sqrt{\sum_i g_i^2}$
2. 如果 $||\mathbf{g}||_2 > \text{max\_norm}$，则等比例缩小所有梯度：

$$\mathbf{g} \leftarrow \frac{\text{max\_norm}}{||\mathbf{g}||_2} \cdot \mathbf{g}$$

3. 如果 $||\mathbf{g}||_2 \leq \text{max\_norm}$，梯度不变

为什么 `max_norm=1.0`？
- 这是业界经验值，GPT-2/3、LLaMA 等主流模型都使用 1.0
- 直觉：将梯度的"步长"限制在单位球内，防止任何一步更新过大
- 实际选择范围通常在 0.5-5.0，1.0 是最常用的默认值
- 与学习率配合：`max_norm=1.0` 意味着单步参数变化量的上界约为 `lr * 1.0`

梯度裁剪 vs 梯度归一化：
- **裁剪**：只在梯度过大时缩放，正常梯度不受影响 → 更常用
- **归一化**：总是将梯度缩放到固定范数 → 可能影响小梯度的更新

**MiniMind 中的实现：**

```python
# 先 unscale 还原梯度真实值
scaler.unscale_(optimizer)
# 按全局范数裁剪
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
```

`clip_grad_norm_` 返回裁剪前的全局范数值，可以用于监控训练状态——如果范数持续远大于 `max_norm`，说明训练不稳定，需要调低学习率。

**追问方向：**
- 除了 clip by norm，还有 clip by value，两者的区别？（clip by value 独立裁剪每个梯度分量，不保持方向；clip by norm 保持梯度方向）
- 梯度裁剪会不会影响收敛？（理论上会改变优化轨迹，但实践中利远大于弊，因为爆炸的梯度方向本身不可靠）

## Q7: 梯度消失和梯度爆炸在 LLM 训练中如何表现？如何排查？

**标准答案：**

**梯度爆炸的表现：**
- Loss 突然变为 NaN 或 Inf
- Loss 突然跳升到很大的值
- `clip_grad_norm_` 返回的梯度范数持续很大（如 > 100）
- 模型输出变为垃圾文本

**梯度消失的表现：**
- Loss 下降极慢或完全不下降
- 较早层的梯度范数远小于较晚层
- 模型生成的文本总是重复相同的模式

排查方法：
1. **监控梯度范数**：记录每步 `clip_grad_norm_` 的返回值，观察趋势
2. **分层检查梯度**：对比不同层的梯度范数，看是否有显著衰减/膨胀
3. **监控 loss 曲线**：loss 震荡或发散通常是梯度爆炸；loss 平坦通常是梯度消失
4. **检查权重分布**：权重值是否过大或过小

LLM 的应对机制（大部分已内建到架构中）：
- **Pre-RMSNorm**：先归一化再计算，稳定各层的输入分布
- **残差连接**：梯度可以"跳过"中间层直达浅层，缓解消失
- **梯度裁剪**：直接限制梯度范数上界
- **学习率 warmup**：避免训练初期大梯度导致参数剧烈变动

**MiniMind 中的实现：**

MiniMind 采用了多层防御：

```python
# 1. 架构层面：Pre-RMSNorm + 残差连接（model_minimind.py）
#    每个 Transformer Block 内部：x = x + attention(rmsnorm(x))

# 2. 训练层面：梯度裁剪
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# 3. 学习率层面：warmup + cosine decay
def get_lr(current_step, warmup_steps, max_steps, max_lr, min_lr):
    if current_step < warmup_steps:
        return max_lr * current_step / warmup_steps
    decay_ratio = (current_step - warmup_steps) / (max_steps - warmup_steps)
    coeff = 0.5 * (1.0 + math.cos(math.pi * decay_ratio))
    return min_lr + coeff * (max_lr - min_lr)
```

MiniMind 的 loss 曲线正常模式为：快速下降 → 缓慢下降 → 趋于平稳。如果出现异常模式（突然上升、持续震荡），首先检查学习率和数据质量。

**追问方向：**
- 为什么 Transformer 比 RNN 更不容易出现梯度消失？（残差连接 + 注意力机制提供直接路径）
- Pre-Norm vs Post-Norm 对训练稳定性的影响？（Pre-Norm 梯度更稳定，是目前 LLM 的主流选择）

## Q8: 为什么预训练的 accumulation_steps=8 而 SFT 通常不需要那么大？

**标准答案：**

核心原因在于两个阶段对 batch size 的需求不同：

**预训练需要大 batch 的原因：**
1. **数据多样性**：预训练数据覆盖广泛的语料（百科、新闻、代码等），需要每个 batch 包含足够多样的样本，才能得到稳定的梯度估计
2. **loss 景观平坦**：预训练初期 loss 景观比较平坦，小 batch 带来的梯度噪声可能导致优化方向不稳定
3. **训练效率**：大 batch 可以更好地利用 GPU 的并行能力，提高每步的有效计算量
4. **学习率可以更大**：大 batch 对应更稳定的梯度，可以使用更大的学习率加速收敛（linear scaling rule）

MiniMind 预训练：`batch_size=32 × accumulation_steps=8 = 等效 256`

**SFT 不需要大 accumulation 的原因：**
1. **数据分布更集中**：SFT 数据都是指令-回答对，分布比预训练数据窄得多，小 batch 就能得到稳定的梯度
2. **微调而非从零学习**：模型已有预训练知识，参数只需小幅调整，梯度信噪比较高
3. **学习率更小**：SFT 用 `lr=1e-5`（预训练 `lr=5e-4`），小学习率本身就降低了对梯度精度的要求
4. **过拟合风险**：SFT 数据量通常较小，过大的等效 batch size 可能导致每个 epoch 的更新步数太少，优化不充分

MiniMind SFT：`batch_size=16`，无需梯度累积

**MiniMind 中的实现：**

```python
# 预训练配置
# lr=5e-4, batch_size=32, accumulation_steps=8, max_seq_len=340
loss = loss / args.accumulation_steps
scaler.scale(loss).backward()
if step % args.accumulation_steps == 0:
    scaler.step(optimizer)

# SFT 配置
# lr=1e-5, batch_size=16, max_seq_len=768
# 无需梯度累积，直接每步更新
masked_loss.backward()
optimizer.step()
optimizer.zero_grad()
```

注意 SFT 的 `max_seq_len=768` 比预训练的 `340` 更大（因为包含 prompt + response），这本身就会消耗更多显存，进一步限制了 batch_size 的上限。

**追问方向：**
- 如果 SFT 数据量很大（如百万级），是否也需要梯度累积？（可能需要，取决于显存和期望的等效 batch size）
- Linear Scaling Rule 是什么？（batch size 翻倍时，学习率也应线性翻倍；但此规则在超大 batch 下会失效）

## Q9: Loss 不收敛的常见原因和排查方法？

**标准答案：**

Loss 不收敛的常见原因（按排查优先级排列）：

1. **学习率问题**（最常见）
   - 学习率太大 → loss 震荡或发散
   - 学习率太小 → loss 下降极慢
   - 没有 warmup → 初期梯度过大导致参数"飞掉"

2. **数据问题**
   - 数据有脏样本（空文本、超长序列、编码错误）
   - label 和 input 不对应
   - padding token 参与了 loss 计算（应使用 `ignore_index`）

3. **梯度问题**
   - 梯度爆炸 → 需要梯度裁剪
   - 梯度累积忘记除以 `accumulation_steps`
   - `optimizer.zero_grad()` 调用位置错误

4. **精度问题**
   - FP16 下梯度下溢 → 使用 GradScaler
   - BF16 精度不够 → 检查 loss 计算是否在 FP32 下进行

5. **模型问题**
   - 权重初始化不当（全零初始化、方差过大）
   - Weight Tying 后某些梯度被覆盖

排查方法：
```
1. 检查 loss 曲线形态（震荡 → lr 大；平坦 → lr 小）
2. 打印梯度范数（每步 clip_grad_norm_ 的返回值）
3. 缩小数据到 10 条，验证能否过拟合
4. 关闭 AMP，用纯 FP32 对比
5. 逐步恢复组件，定位问题来源
```

**MiniMind 中的实现：**

MiniMind 中预训练 loss 的正常曲线：

```
Loss
  |
8 |*
  | *
6 |  *
  |   **
4 |     ***
  |        ****
2 |            **********
  |                      **********
  +------------------------------------→ Steps
```

异常诊断：
- **loss 突然上升**：学习率太大，或数据中有噪声 batch
- **loss 持续不下降**：学习率太小，或模型太小无法拟合
- **loss 震荡剧烈**：batch size 太小，梯度估计噪声大

MiniMind 中 loss 使用 `F.cross_entropy` 并设置 `ignore_index=-100` 排除 padding token：

```python
loss = F.cross_entropy(
    logits.view(-1, vocab_size),
    labels.view(-1),
    ignore_index=-100
)
```

## Q10: DDP（Distributed Data Parallel）的原理？

**标准答案：**

DDP 是 PyTorch 中最常用的数据并行策略，核心思想：**每张 GPU 持有完整的模型副本，各自处理不同的数据，然后同步梯度**。

工作流程：
1. **初始化**：将模型完整复制到每张 GPU 上，确保所有副本初始参数相同
2. **数据分片**：通过 `DistributedSampler` 将数据集均分给各 GPU，每个 GPU 处理不同的 mini-batch
3. **前向传播**：各 GPU 独立进行前向传播
4. **反向传播 + AllReduce**：反向传播时，DDP 会注册 hook，在梯度计算完成后立即对该参数的梯度进行 AllReduce（求平均），与反向传播重叠执行
5. **参数更新**：所有 GPU 用同步后的梯度独立更新参数（因为梯度相同、参数相同、优化器状态相同，更新结果一定相同）

关键设计：
- **Bucket 机制**：DDP 将参数梯度分成多个 bucket（默认 25MB），按 bucket 粒度做 AllReduce，与反向传播重叠以隐藏通信开销
- **梯度同步与计算重叠**：反向传播从最后一层开始计算梯度，已算完的层的梯度可以立即通信，无需等所有层算完

**MiniMind 中的实现：**

MiniMind 支持 DDP 多卡训练：

```python
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

# 初始化分布式环境
dist.init_process_group(backend='nccl')
local_rank = int(os.environ['LOCAL_RANK'])
torch.cuda.set_device(local_rank)

# 包装模型
model = model.to(local_rank)
model = DDP(model, device_ids=[local_rank])

# 数据 Sampler
sampler = DistributedSampler(dataset)
dataloader = DataLoader(dataset, sampler=sampler)
```

启动命令：

```bash
# 单卡
python trainer/train_pretrain.py

# DDP 多卡（2卡）
torchrun --nproc_per_node=2 trainer/train_pretrain.py
```

注意 `torchrun` 会自动设置 `LOCAL_RANK`、`RANK`、`WORLD_SIZE` 等环境变量，无需手动指定。

## Q11: DDP 中的 AllReduce 操作是什么？通信开销如何计算？

**标准答案：**

AllReduce 是一种集合通信操作，目标是：**让所有节点都得到所有节点上某个值的聚合结果**（在 DDP 中是梯度的平均值）。

常见实现——Ring AllReduce：
1. N 个 GPU 排成环形
2. 每个 GPU 将数据切成 N 份
3. 第一轮（Scatter-Reduce）：每个 GPU 向后传递一份数据并累加，经过 N-1 轮后每个 GPU 上有一份完整的部分和
4. 第二轮（AllGather）：将部分和广播到所有 GPU，再经 N-1 轮

通信开销计算：

设模型参数量为 $P$，每个参数的梯度占 $B$ 字节，GPU 数量为 $N$：

- Ring AllReduce 总通信量：$2 \times (N-1)/N \times P \times B$
- 当 $N$ 较大时，约为 $2PB$，与 GPU 数量近似无关
- 瓶颈在于最慢的链路带宽

MiniMind 64M 参数，FP32 梯度：
$$\text{通信量} \approx 2 \times 64 \times 10^6 \times 4 \text{ bytes} = 512 \text{ MB}$$

在 NVLink（600 GB/s）下约 0.85ms，在 PCIe 4.0 x16（32 GB/s）下约 16ms。

**MiniMind 中的实现：**

MiniMind 使用 NCCL 后端进行 AllReduce，NCCL 是 NVIDIA 提供的高性能集合通信库，自动选择最优的通信拓扑：

```python
dist.init_process_group(backend='nccl')
```

DDP 在底层自动调用 AllReduce，用户无需手动操作。梯度通信与反向传播通过 bucket 机制重叠：

```python
model = DDP(model, device_ids=[local_rank])
# DDP 自动在 backward hook 中触发 AllReduce
```

**追问方向：**
- Ring AllReduce 和 Tree AllReduce 的区别？（Tree 在跨节点场景下延迟更低，适合多机训练）
- 如何减少 AllReduce 的通信开销？（梯度压缩、通信与计算重叠、混合精度——FP16 梯度通信量减半）

## Q12: 数据并行 vs 模型并行 vs 流水线并行的区别和适用场景？

**标准答案：**

| 并行策略 | 划分对象 | 通信内容 | 适用场景 |
|----------|----------|----------|----------|
| 数据并行 (DP) | 数据 | 梯度 | 模型能放进单张 GPU |
| 模型并行 (TP) | 模型层内（如矩阵分块） | 激活值 | 单层参数过大 |
| 流水线并行 (PP) | 模型层间 | 层间激活值 | 模型层数很多 |

**数据并行（Data Parallelism）：**
- 每个 GPU 持有完整模型，处理不同数据
- 通信：AllReduce 同步梯度
- 优点：实现简单，扩展性好
- 缺点：每个 GPU 都要存完整模型，显存不够用时无法使用
- 代表：PyTorch DDP

**模型并行（Tensor Parallelism）：**
- 将单层的权重矩阵切分到多个 GPU（如 $W$ 按列切为 $[W_1, W_2]$）
- 通信：每层前向和反向都需要 AllReduce 或 AllGather 同步激活值
- 优点：可以训练单 GPU 放不下的超大模型
- 缺点：通信频率高，对 GPU 间带宽要求极高（通常要 NVLink）
- 代表：Megatron-LM

**流水线并行（Pipeline Parallelism）：**
- 将模型的不同层放到不同 GPU
- 通信：层间传递激活值和梯度
- 优点：通信量小（只在层边界传递一次）
- 缺点：存在 "bubble"（流水线空泡），GPU 利用率不满
- 代表：GPipe, PipeDream

实际大模型训练通常采用 **3D 并行**：DP + TP + PP 组合。

**MiniMind 中的实现：**

MiniMind 作为小模型（26M/64M/218M），单卡即可完整放下，因此采用最简单的数据并行（DDP）：

```python
model = DDP(model, device_ids=[local_rank])
sampler = DistributedSampler(dataset)
```

对于 MiniMind 的规模，数据并行是最优选择——模型并行和流水线并行引入的通信开销和实现复杂度不划算。

**追问方向：**
- 什么规模的模型开始需要模型并行？（粗略经验：单卡放不下时，通常 >10B 参数）
- 3D 并行中，TP、PP、DP 的优先级是什么？（通常先在同节点内 TP，再跨节点 PP，最后 DP）

## Q13: DeepSpeed ZeRO 的三个阶段分别优化了什么？

**标准答案：**

ZeRO（Zero Redundancy Optimizer）是 DeepSpeed 提出的优化策略，核心思想是**消除数据并行中的冗余显存占用**。

训练时的显存占用由三部分组成（以 FP32 + Adam 为例，参数量 $\Psi$）：
- 模型参数：$4\Psi$ 字节（FP32）
- 梯度：$4\Psi$ 字节（FP32）
- 优化器状态：$12\Psi$ 字节（Adam 需要存 FP32 参数副本 + 一阶矩 + 二阶矩）

标准 DDP 中，每张 GPU 都完整存储以上所有内容 = $20\Psi$ 字节/卡。

**ZeRO Stage 1 — 分片优化器状态：**
- 每张 GPU 只存 $1/N$ 的优化器状态
- 显存：$4\Psi + 4\Psi + 12\Psi/N$
- 通信量：与 DDP 相同

**ZeRO Stage 2 — 分片优化器状态 + 梯度：**
- 每张 GPU 只存 $1/N$ 的优化器状态和梯度
- 显存：$4\Psi + 4\Psi/N + 12\Psi/N$
- 通信量：与 DDP 相同

**ZeRO Stage 3 — 分片一切（参数 + 梯度 + 优化器状态）：**
- 每张 GPU 只存 $1/N$ 的所有内容
- 显存：$(4 + 4 + 12)\Psi / N = 20\Psi/N$
- 通信量：增加约 50%（前向传播时需要 AllGather 收集其他 GPU 的参数）

| 阶段 | 分片内容 | 显存节省 | 通信开销 |
|------|----------|----------|----------|
| Stage 1 | 优化器状态 | ~4x（8卡） | 与 DDP 相同 |
| Stage 2 | 优化器状态 + 梯度 | ~8x（8卡） | 与 DDP 相同 |
| Stage 3 | 全部 | ~N 倍 | 增加 ~50% |

**MiniMind 中的实现：**

MiniMind 文档中提及支持 DeepSpeed 训练：

```bash
deepspeed trainer/train_pretrain.py --deepspeed ds_config.json
```

对于 MiniMind 的小模型规模（64M 参数），通常使用 ZeRO Stage 1 或 Stage 2 即可。Stage 3 更适合百亿级以上参数的大模型。

**追问方向：**
- ZeRO Stage 3 和 FSDP 有什么关系？（FSDP 是 PyTorch 原生的 ZeRO-3 实现）
- ZeRO-Offload 和 ZeRO-Infinity 分别做了什么？（Offload 将优化器状态卸载到 CPU；Infinity 进一步卸载到 NVMe SSD）

## Q14: FSDP（Fully Sharded Data Parallel）的原理？

**标准答案：**

FSDP 是 PyTorch 原生实现的全分片数据并行，本质上是 ZeRO Stage 3 的 PyTorch 版本。

核心思想：**每个 GPU 只保留 $1/N$ 的模型参数、梯度和优化器状态，需要时通过通信临时收集**。

工作流程：
1. **初始化分片**：将模型参数分片到 N 个 GPU，每个 GPU 只存 $1/N$
2. **前向传播**：
   - 当前层计算前，AllGather 收集完整参数
   - 计算完成后，释放非本地分片（可选）
3. **反向传播**：
   - 当前层反向计算前，AllGather 收集完整参数
   - 计算梯度后，ReduceScatter 将梯度分片回各 GPU
   - 释放完整参数
4. **参数更新**：每个 GPU 用本地的梯度分片更新本地的参数分片

FSDP vs DDP：

| 特性 | DDP | FSDP |
|------|-----|------|
| 参数存储 | 完整（每卡） | 分片（1/N） |
| 显存占用 | 高 | 低 |
| 通信量 | AllReduce | AllGather + ReduceScatter |
| 通信频率 | 仅反向传播 | 前向 + 反向都需要 |
| 实现复杂度 | 低 | 较高 |

**MiniMind 中的实现：**

MiniMind 当前使用 DDP 而非 FSDP，因为模型规模较小（64M），单卡完全可以放下完整模型。FSDP 的额外通信开销在小模型上不划算：

```python
# MiniMind 使用 DDP
model = DDP(model, device_ids=[local_rank])

# 如果要改用 FSDP，只需替换为：
# from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
# model = FSDP(model)
```

对于 MiniMind-218M 这样稍大的变体，如果在显存紧张的消费级 GPU（如 8GB）上多卡训练，FSDP 可以带来显著的显存收益。

**追问方向：**
- FSDP 的 `sharding_strategy` 有哪些选项？（FULL_SHARD = ZeRO-3, SHARD_GRAD_OP = ZeRO-2, NO_SHARD = DDP）
- FSDP 的 `auto_wrap_policy` 如何选择？（通常按 Transformer Block 粒度包装）

## Q15: MiniMind 如何实现 DDP 多卡训练？

**标准答案：**

MiniMind 的 DDP 实现涵盖以下关键步骤：

1. **进程组初始化**：使用 NCCL 后端初始化分布式环境
2. **模型分发**：将模型移到对应 GPU 并用 DDP 包装
3. **数据分片**：使用 DistributedSampler 保证每个 GPU 处理不同数据
4. **梯度同步**：DDP 自动在反向传播时同步梯度
5. **日志管理**：只在 rank 0 进程打印日志和保存 checkpoint

**MiniMind 中的实现：**

完整的 DDP 设置流程：

```python
import os
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler

# 1. 初始化分布式环境
dist.init_process_group(backend='nccl')
local_rank = int(os.environ['LOCAL_RANK'])
torch.cuda.set_device(local_rank)

# 2. 模型准备
model = MiniMindForCausalLM(config).to(local_rank)
model = DDP(model, device_ids=[local_rank])

# 3. 数据准备（DistributedSampler 自动分片）
sampler = DistributedSampler(dataset)
dataloader = DataLoader(dataset, sampler=sampler, batch_size=batch_size)

# 4. 训练循环
for epoch in range(num_epochs):
    sampler.set_epoch(epoch)  # 每个 epoch 重新 shuffle
    for step, batch in enumerate(dataloader):
        # ... 前向、反向、更新 ...
        pass

    # 5. 只在 rank 0 保存 checkpoint
    if local_rank == 0:
        torch.save(model.module.state_dict(), 'checkpoint.pt')

# 6. 清理
dist.destroy_process_group()
```

启动方式：

```bash
torchrun --nproc_per_node=2 trainer/train_pretrain.py
```

注意事项：
- `sampler.set_epoch(epoch)` 确保每个 epoch 的 shuffle 顺序不同
- 保存模型时用 `model.module.state_dict()`（去掉 DDP 的包装层）
- `dist.barrier()` 用于多卡间的同步点

**追问方向：**

## Q16: 断点续训需要保存哪些状态？MiniMind 保存了什么？

**标准答案：**

完整的断点续训需要保存以下状态，确保恢复后训练结果与不中断完全一致：

| 状态 | 为什么需要 | 不保存的后果 |
|------|------------|------------|
| 模型参数 (`model.state_dict()`) | 核心状态 | 从头训练 |
| 优化器状态 (`optimizer.state_dict()`) | Adam 的动量 m、v | 优化器预热丢失，收敛变慢 |
| 学习率调度器状态 | 恢复正确的 lr | lr 从初始值重新开始 |
| GradScaler 状态 (`scaler.state_dict()`) | scale factor 和 growth tracker | AMP 状态重置 |
| 训练进度 (epoch, step) | 知道从哪里继续 | 重复训练已见数据 |
| 随机数种子 (RNG states) | 数据顺序可复现 | 数据顺序改变 |
| wandb run ID | 续接同一条日志曲线 | 创建新的 run |

**MiniMind 中的实现：**

MiniMind 保存的 checkpoint 包含：

```python
# 保存
torch.save({
    'model': model.state_dict(),
    'optimizer': optimizer.state_dict(),
    'scaler': scaler.state_dict(),
    'epoch': epoch,
    'step': global_step,
    'wandb_id': wandb_run_id,
}, 'checkpoint.pt')

# 恢复
ckpt = torch.load('checkpoint.pt')
model.load_state_dict(ckpt['model'])
optimizer.load_state_dict(ckpt['optimizer'])
scaler.load_state_dict(ckpt['scaler'])
start_epoch = ckpt['epoch']
start_step = ckpt['step']
wandb.init(id=ckpt['wandb_id'], resume='allow')
```

MiniMind 保存了 6 项核心状态。注意 `wandb_id` 的保存让中断恢复后的日志能续接到同一条曲线上，避免产生多个零碎的 run。

**追问方向：**
- 如果只保存了 model state_dict 而没保存 optimizer state_dict，恢复训练会怎样？（Adam 的动量 m 和 v 归零，前几百步相当于重新预热优化器，loss 可能短暂上升）
- checkpoint 文件太大怎么办？（只保存关键状态；使用 ZeRO Stage 3 分片保存；间隔保存而非每步保存）

## Q17: 跨 GPU 数量恢复训练的注意事项？

**标准答案：**

从 N 卡训练的 checkpoint 恢复到 M 卡训练（N ≠ M）时，需要注意以下问题：

1. **模型参数**：
   - DDP 模式下模型参数与 GPU 数量无关，可以直接加载
   - 注意 DDP 包装后的 key 前缀：保存时用 `model.module.state_dict()`，加载时先加载到单卡模型再包装 DDP

2. **优化器状态**：
   - Adam 等优化器的状态（m, v）与参数一一对应，可以直接加载
   - 但 FSDP/ZeRO-3 模式下，优化器状态是分片的，需要先合并再重新分片

3. **学习率调度器**：
   - 如果 lr scheduler 基于 step 数，需要调整：不同 GPU 数量下，相同数据量的 step 数不同
   - `total_steps_new = total_steps_old × N / M`

4. **数据加载**：
   - `DistributedSampler` 自动适配 GPU 数量
   - 但如果保存了 DataLoader 的迭代位置，需要重新计算

5. **等效 batch size 变化**：
   - `effective_batch = batch_per_gpu × num_gpus × accumulation_steps`
   - GPU 数量变化后需要调整 `batch_per_gpu` 或 `accumulation_steps` 保持等效 batch size 不变
   - 或按 linear scaling rule 调整学习率

**MiniMind 中的实现：**

MiniMind 使用 DDP，checkpoint 中保存的是完整模型参数（非分片），跨 GPU 数量恢复相对简单：

```python
# 保存时：保存完整参数（去掉 DDP 包装）
state_dict = model.module.state_dict()
torch.save({'model': state_dict, ...}, 'checkpoint.pt')

# 恢复时：先加载到单卡，再包装 DDP
model = MiniMindForCausalLM(config)
model.load_state_dict(torch.load('checkpoint.pt')['model'])
model = model.to(local_rank)
model = DDP(model, device_ids=[local_rank])
```

如果从 1 卡切换到 2 卡，等效 batch size 翻倍，建议相应调整 `accumulation_steps` 或学习率：

```python
# 1卡：batch_size=32, accumulation_steps=8 → effective=256
# 2卡：batch_size=32, accumulation_steps=4 → effective=256 (保持不变)
```

**追问方向：**
- FSDP/ZeRO-3 的分片 checkpoint 如何跨 GPU 数量恢复？（需要先合并分片为完整 state_dict，再重新分片）
- 跨不同型号 GPU 恢复（如从 A100 到 3090）有什么需要注意的？（BF16 支持情况、显存大小影响 batch size）

## Q18: 模型权重保存为 FP16 的实现和好处？

**标准答案：**

训练时模型权重是 FP32（32 位），但推理和存储时可以转为 FP16（16 位），好处：

1. **存储空间减半**：FP32 每个参数 4 字节 → FP16 只需 2 字节
   - MiniMind 64M 参数：FP32 = 256MB，FP16 = 128MB
2. **推理加速**：FP16 推理速度通常是 FP32 的 2 倍以上
3. **部署友好**：更小的模型更容易在消费级硬件和移动端部署
4. **精度损失可忽略**：推理阶段不涉及梯度计算和参数更新，FP16 精度对最终输出影响极小

**MiniMind 中的实现：**

MiniMind 使用字典推导式将权重逐一转换为 FP16 并移到 CPU：

```python
state_dict = model.state_dict()
# 遍历所有参数，转为半精度并移至 CPU
half_state_dict = {k: v.half().cpu() for k, v in state_dict.items()}
torch.save(half_state_dict, 'model_fp16.pt')
```

这个实现的细节：
- `.half()` 将 FP32 转为 FP16
- `.cpu()` 将参数从 GPU 移到 CPU 内存，避免保存时占用 GPU 显存

## Q20: 权重初始化策略有哪些？为什么初始化很重要？

**标准答案：**

权重初始化决定了训练的起点，直接影响训练能否收敛以及收敛速度。

**为什么初始化很重要？**
- **全零初始化**：所有神经元计算相同的值，梯度也相同，无法打破对称性 → 训练失败
- **随机但方差过大**：激活值会层层放大，导致梯度爆炸
- **随机但方差过小**：激活值会层层衰减，导致梯度消失

常见初始化策略：

| 策略 | 公式 | 适用场景 |
|------|------|----------|
| Xavier (Glorot) | $W \sim \mathcal{U}(-\sqrt{6/(n_{in}+n_{out})}, +\sqrt{6/(n_{in}+n_{out})})$ | Sigmoid/Tanh 激活 |
| Kaiming (He) | $W \sim \mathcal{N}(0, \sqrt{2/n_{in}})$ | ReLU 系列激活 |
| 截断正态分布 | $W \sim \text{TruncNormal}(0, 0.02)$ | Transformer 模型 |
| 小常数 | $W \sim \mathcal{N}(0, 0.01)$ | 最后一层/输出层 |

LLM 中的常见做法：
- GPT-2/3：`N(0, 0.02)`，残差路径的最后一层额外除以 $\sqrt{2N}$（N 为层数）
- LLaMA：`N(0, 0.02)` 截断正态分布

**MiniMind 中的实现：**

MiniMind 的模型架构与 LLaMA 一致，使用 PyTorch 的默认初始化策略：

```python
class MiniMindForCausalLM(nn.Module):
    def __init__(self, config):
        super().__init__()
        self.model = MiniMindModel(config)
        self.lm_head = nn.Linear(config.dim, config.vocab_size, bias=False)
        # Weight Tying：lm_head 和 embed_tokens 共享权重
        self.lm_head.weight = self.model.embed_tokens.weight
```

- `nn.Linear`：默认使用 Kaiming Uniform 初始化
- `nn.Embedding`：默认使用 `N(0, 1)` 初始化
- Weight Tying 意味着 `lm_head` 和 `embed_tokens` 的权重是同一个张量，初始化只需做一次

**追问方向：**
- Weight Tying 对初始化有什么影响？（共享权重只初始化一次，但需确保初始化方式同时适用于 embedding 和 linear 输出层）
- 为什么 GPT 系列的残差路径要除以 $\sqrt{2N}$？（补偿残差连接导致的方差放大：每层残差加法使方差加倍，N 层后方差变为 2N 倍）

## Q21: Xavier/Kaiming 初始化的适用场景？

**标准答案：**

两种初始化的核心目标相同：**保持前向传播中激活值的方差和反向传播中梯度的方差在各层间稳定不变**。区别在于对激活函数的假设。

**Xavier 初始化（2010, Glorot）：**
- 假设：激活函数是线性的或关于原点对称（如 Sigmoid、Tanh）
- 推导：令 $\text{Var}(y) = \text{Var}(x)$，得到 $\text{Var}(W) = \frac{2}{n_{in} + n_{out}}$
- 均匀分布版本：$W \sim \mathcal{U}(-\sqrt{6/(n_{in}+n_{out})}, +\sqrt{6/(n_{in}+n_{out})})$
- 正态分布版本：$W \sim \mathcal{N}(0, 2/(n_{in}+n_{out}))$

**Kaiming 初始化（2015, He）：**
- 假设：激活函数是 ReLU（会"杀死"一半的神经元）
- 推导：ReLU 使约一半的输出为零，方差减半，因此补偿系数为 2：$\text{Var}(W) = \frac{2}{n_{in}}$
- 正态分布版本：$W \sim \mathcal{N}(0, 2/n_{in})$

在 LLM 中：
- Transformer 使用 GELU/SiLU/SwiGLU 等激活函数，严格来说都不完全满足 Xavier 或 Kaiming 的假设
- 实践中**截断正态分布 `N(0, 0.02)`** 更常用，这是 GPT 系列的经验选择
- RMSNorm 的归一化效果可以补偿初始化不完美带来的方差漂移

**MiniMind 中的实现：**

MiniMind 使用 SwiGLU 作为 FFN 的激活函数：

```python
# MiniMind 的 FFN（SwiGLU）
class FeedForward(nn.Module):
    def __init__(self, config):
        super().__init__()
        hidden_dim = int(config.dim * config.ffn_dim_multiplier)
        self.w1 = nn.Linear(config.dim, hidden_dim, bias=False)
        self.w2 = nn.Linear(hidden_dim, config.dim, bias=False)
        self.w3 = nn.Linear(config.dim, hidden_dim, bias=False)

    def forward(self, x):
        return self.w2(F.silu(self.w1(x)) * self.w3(x))
```

`nn.Linear` 默认使用 Kaiming Uniform 初始化（`mode='fan_in'`），这对 SwiGLU 来说是合理的近似。Pre-RMSNorm 的存在进一步降低了对初始化精确性的依赖。

**追问方向：**
- 如果把 Kaiming 初始化换成 Xavier，LLM 训练会有什么影响？（影响不大，RMSNorm 可以补偿；但在深层网络中 Kaiming 通常更稳定）
- LLaMA 的具体初始化方式是什么？（embedding 用 `N(0, 1)`，其他线性层用截断正态分布）

## Q22: 学习率 Warmup 对训练稳定性的影响？

**标准答案：**

Warmup 是指在训练的前若干步中，让学习率从 0（或很小的值）**线性增长**到目标学习率。

**为什么需要 Warmup？**

1. **初期梯度不可靠**：模型参数刚初始化，梯度方向高度随机。大学习率 + 随机梯度 = 参数可能被"推到"不好的区域
2. **Adam 需要预热**：Adam 优化器的自适应学习率依赖于一阶矩（m）和二阶矩（v）的估计，初始几步估计严重偏差（虽然有偏差修正，但效果有限）
3. **Loss Landscape 稳定性**：大模型的 loss 景观在初始化点附近可能有很陡的区域，warmup 帮助模型平稳地进入平缓区域

Warmup 的效果：
- 没有 warmup：loss 可能在初期剧烈震荡，甚至发散
- 有 warmup：loss 平稳下降，训练更稳定

典型的 warmup 步数：总训练步数的 1-5%（如 1000-2000 步）

**MiniMind 中的实现：**

MiniMind 实现了线性 warmup + 余弦退火的学习率调度：

```python
def get_lr(current_step, warmup_steps, max_steps, max_lr, min_lr):
    # Warmup 阶段：线性增长
    if current_step < warmup_steps:
        return max_lr * current_step / warmup_steps
    # 训练完成后保持最小学习率
    if current_step > max_steps:
        return min_lr
    # Cosine 退火阶段
    decay_ratio = (current_step - warmup_steps) / (max_steps - warmup_steps)
    coeff = 0.5 * (1.0 + math.cos(math.pi * decay_ratio))
    return min_lr + coeff * (max_lr - min_lr)
```

学习率曲线示意：

```
学习率
  |    /\
  |   /  \
  |  /    \
  | /      \____
  |/            \
  +-------------→ 训练步数
  ↑warmup  ↑cosine decay
```

MiniMind 的余弦退火也可以表达为用户提供的公式形式：`lr * (0.1 + 0.45 * (1 + cos(π * step / total)))`，其中 0.1 对应 `min_lr/max_lr`，0.45 = 0.5 × (1 - min_lr/max_lr)。

**追问方向：**
- 除了线性 warmup，还有哪些 warmup 策略？（指数 warmup、多项式 warmup；线性最常用）
- Warmup 步数如何选择？（经验值 + 实验；一般占总步数的 1-5%；大模型需要更长的 warmup）

## Q23: 训练 Loss 出现 NaN/Inf 的常见原因和解决方法？

**标准答案：**

NaN/Inf 出现的常见原因：

**1. 数值溢出（Overflow）——最常见**
- FP16 的最大值仅约 65504，中间计算结果超过此值变为 Inf
- Inf 参与后续运算产生 NaN（如 Inf - Inf = NaN, 0 × Inf = NaN）
- 解决：使用 BF16（范围与 FP32 一致）或启用 GradScaler

**2. 学习率过大**
- 参数更新步长过大，权重值增长失控
- 解决：降低学习率；使用 warmup；使用梯度裁剪

**3. Loss 计算中的除零**
- `log(0)` = -Inf
- SFT 中 `loss_mask.sum()` 为 0 时出现除零
- 解决：添加 eps（如 `loss_mask.sum() + 1e-8`）

**4. 数据质量问题**
- 输入包含 NaN 或异常值
- Token ID 超出词表范围（导致 embedding lookup 越界）
- 解决：数据预处理时做清洗和校验

**5. 模型架构问题**
- 激活值过大（softmax 前的 logits 过大）→ 使用 scaled dot-product attention
- 权重初始化方差过大

排查步骤：

```python
# 1. 检测 NaN 出现的位置
torch.autograd.set_detect_anomaly(True)

# 2. 在关键位置插入检查
assert not torch.isnan(loss), f"NaN loss at step {step}"
assert not torch.isinf(loss), f"Inf loss at step {step}"

# 3. 检查输入数据
for batch in dataloader:
    assert not torch.isnan(batch).any()
    break
```

**MiniMind 中的实现：**

MiniMind 通过以下机制预防 NaN/Inf：

```python
# 1. 使用 BF16 避免 FP16 的溢出问题
with autocast(dtype=torch.bfloat16):
    res = model(input_ids, labels=labels)
    loss = res.loss

# 2. GradScaler 检测到 inf/nan 时自动跳过更新
scaler.step(optimizer)  # 内部检查梯度是否有 inf/nan
scaler.update()         # 自动调整 scale factor

# 3. 梯度裁剪防止梯度爆炸导致的权重溢出
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# 4. Loss 计算中使用 ignore_index 避免无效 token
loss = F.cross_entropy(logits.view(-1, V), labels.view(-1), ignore_index=-100)
```

- 如何实现自动从最近正常 checkpoint 恢复的 "loss spike recovery"？

## Q25: 训练数据的格式设计？JSONL vs Parquet vs Arrow？

**标准答案：**

| 格式 | 特点 | 适用场景 |
|------|------|----------|
| **JSONL** | 每行一个 JSON，可读性好，流式读取 | 小到中型数据集，快速迭代 |
| **Parquet** | 列式存储，压缩率高，查询快 | 大型数据集，需要列过滤 |
| **Arrow** | 内存映射，零拷贝读取 | HuggingFace datasets 后端 |
| **TFRecord** | TensorFlow 生态，支持流式 | TF 项目 |
| **MDS** | MosaicML 流式格式 | 超大规模分布式训练 |

**JSONL 的优势（MiniMind 的选择）：**
1. **人类可读**：可以直接打开查看数据质量
2. **流式处理**：逐行读取，不需要一次性加载到内存
3. **灵活性高**：字段可以随意添加，不需要预定义 schema
4. **调试友好**：出问题时可以直接定位到具体行号

JSONL 的缺点：
- 解析速度比二进制格式慢
- 不支持压缩读取（需要先解压）
- 文件体积较大

实际工程建议：
- 数据量 < 10GB → JSONL 足够
- 数据量 10-100GB → Parquet 更合适
- 数据量 > 100GB → Arrow/MDS + 流式加载

**MiniMind 中的实现：**

MiniMind 使用 JSONL 格式存储预训练数据，每行包含一个 `text` 字段：

```python
# dataset/pretrain_t2t_mini.jsonl 数据格式
# {"text": "机器学习是人工智能的一个分支..."}
# {"text": "深度学习使用多层神经网络..."}
# {"text": "自然语言处理是计算机科学..."}

class PretrainDataset(Dataset):
    def __init__(self, data_path, tokenizer, max_length):
        self.data = []
        with open(data_path, 'r') as f:
            for line in f:
                self.data.append(json.loads(line))
        self.tokenizer = tokenizer
        self.max_length = max_length

    def __getitem__(self, idx):
        text = self.data[idx]['text']
        token_ids = self.tokenizer.encode(text)
        token_ids = token_ids[:self.max_length]
        return token_ids
```

注意这里将所有数据一次性加载到内存（`self.data`），对于 MiniMind 的小数据集是合理的。大数据集应改用内存映射或流式读取。

## Q26: 数据加载的 DataLoader 配置技巧？num_workers、pin_memory 等

**标准答案：**

`DataLoader` 的关键参数及优化建议：

**`num_workers`（数据加载并行度）：**
- 作用：使用多个子进程并行预取数据
- 经验值：`num_workers = min(CPU 核数, 4 * GPU 数)`
- 设为 0 → 主进程加载数据，GPU 等待数据时空转
- 设太大 → 内存开销增加，进程间通信成本高
- 调优方法：递增测试，直到 GPU 利用率不再提升

**`pin_memory`（锁页内存）：**
- 作用：将数据预先放入锁页内存，加速 CPU→GPU 的数据传输
- 建议：在 GPU 训练时始终设为 `True`
- 原理：锁页内存不会被操作系统换出到磁盘，`cudaMemcpyAsync` 可以直接 DMA 传输

**`prefetch_factor`（预取因子）：**
- 作用：每个 worker 提前加载的 batch 数量
- 默认值：2
- 增大 → 数据预取更充分，减少 GPU 等待，但占更多内存

**`persistent_workers`（持久 worker）：**
- 作用：不在每个 epoch 结束后销毁 worker
- 建议：`True`（避免反复创建进程的开销）

**`drop_last`（丢弃最后不完整 batch）：**
- DDP 训练时建议 `True`，避免不同 GPU 的 batch 大小不一致

**MiniMind 中的实现：**

MiniMind 的 DataLoader 配置：

```python

    batch_size=32,
    shuffle=True,

    collate_fn=collate_fn

# DDP 模式

    dataset,

    num_workers=4,
    pin_memory=True,
    drop_last=True,
    collate_fn=collate_fn
)
```

注意：使用 `DistributedSampler` 时，`shuffle` 参数必须去掉（或设为 False），因为 shuffle 逻辑由 sampler 控制。

**追问方向：**
- `num_workers > 0` 时，Dataset 的 `__getitem__` 需要注意什么？（不能使用不可序列化的对象；文件句柄应在 worker 初始化时打开而非 `__init__`）
- 如何诊断数据加载是否是训练的瓶颈？（检查 GPU 利用率：如果 GPU 利用率波动大或有长时间为 0 的间隔，说明在等数据）

## Q27: 动态长度填充（Dynamic Padding）vs 固定长度截断？

**标准答案：**

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **固定长度截断** | 所有样本 pad/truncate 到 `max_seq_len` | 实现简单，batch 形状固定 | 浪费计算（短样本的 padding 做无用计算） |
| **动态长度填充** | 每个 batch 内 pad 到该 batch 最长序列的长度 | 节省计算量 | 实现更复杂，batch 形状不固定 |

**固定长度截断：**

```python
# 所有样本都 pad/truncate 到 max_seq_len
token_ids = token_ids[:max_seq_len]  # 截断
token_ids = token_ids + [pad_id] * (max_seq_len - len(token_ids))  # 填充
```

**动态长度填充：**

```python
def collate_fn(batch):
    max_len = max(len(seq) for seq in batch)  # 当前 batch 的最大长度
    padded = []
    for seq in batch:
        padded.append(seq + [pad_id] * (max_len - len(seq)))
    return torch.tensor(padded)
```

进阶优化——**按长度分桶（Bucketing）**：
- 将样本按长度排序或分组
- 同一 bucket 内的样本长度相近，减少 padding 浪费
- HuggingFace 的 `group_by_length` 实现了此策略

**MiniMind 中的实现：**

MiniMind 使用固定长度截断策略，这是最简单直接的方式：

```python
# 预训练：max_seq_len=340
token_ids = self.tokenizer.encode(text)
token_ids = token_ids[:self.max_length]

# SFT：max_seq_len=768
input_ids = input_ids[:max_seq_len]
loss_mask = loss_mask[:max_seq_len]
```

对于 MiniMind 的小模型和小数据集，固定长度截断足够高效。`max_seq_len=340`（预训练）的选择覆盖了绝大部分训练样本的长度，padding 浪费不多。

SFT 阶段 `max_seq_len=768` 更大，因为需要容纳完整的 prompt + response 对话。

**追问方向：**
- Packing（序列拼接）是什么？和 Padding 相比的优缺点？（将多个短序列拼接到一个长序列中，消除 padding 浪费；但需要修改 attention mask 防止跨序列注意力）
- 动态 padding 对 Tensor Core 利用率有影响吗？（是的，Tensor Core 对特定的矩阵维度有最佳性能，动态长度可能导致维度不是 8/16 的倍数）

## Q28: 训练数据的洗牌策略？

**标准答案：**

数据洗牌（Shuffle）对训练质量至关重要：

**为什么需要洗牌？**
1. **避免顺序偏差**：如果数据按类别/来源排列，连续 batch 可能来自同一分布，梯度有系统性偏差
2. **改善泛化**：随机采样使每个 batch 更具代表性，梯度方向更接近全局最优
3. **防止"记忆"顺序**：模型可能学到数据的排列顺序而非真正的规律

**洗牌策略层次：**

| 层次 | 方式 | 说明 |
|------|------|------|
| 文件级 | 打乱 shard 文件顺序 | 适用于大规模分布式训练 |
| 样本级 | `DataLoader(shuffle=True)` | 最常用，每个 epoch 重新排列 |
| Buffer 级 | 从固定大小的 buffer 中随机采样 | 适用于流式数据（`IterableDataset`） |

**DDP 下的洗牌注意事项：**
- 使用 `DistributedSampler` 自动处理跨 GPU 的数据分片
- 必须调用 `sampler.set_epoch(epoch)` 确保每个 epoch 的 shuffle 种子不同

**MiniMind 中的实现：**

单卡训练：

```python
dataloader = DataLoader(dataset, batch_size=batch_size, shuffle=True)
```

DDP 多卡训练：

```python
sampler = DistributedSampler(dataset)
dataloader = DataLoader(dataset, sampler=sampler)

for epoch in range(num_epochs):
    sampler.set_epoch(epoch)  # 关键：每 epoch 重新 shuffle
    for batch in dataloader:
        # ...
```

`set_epoch(epoch)` 的原理：`DistributedSampler` 使用 `epoch` 作为随机种子的一部分，不同 epoch 生成不同的 shuffle 顺序。如果不调用，每个 epoch 的数据顺序完全相同。

**追问方向：**
- 洗牌对预训练和 SFT 的影响相同吗？（预训练数据量大，洗牌效果更显著；SFT 数据量小，多 epoch 训练时洗牌更重要，避免过拟合到特定顺序）
- 超大数据集（TB 级）如何高效洗牌？（不可能一次性加载到内存；使用分片 shuffle + buffer shuffle 的两级策略）

## Q29: torch.compile 的原理和使用场景？编译模式的区别？

**标准答案：**

`torch.compile` 是 PyTorch 2.0 引入的模型编译功能，通过将 Python 代码编译为优化后的内核来加速执行。

**原理：**
1. **TorchDynamo**：Python 层的字节码分析，提取计算图
2. **TorchInductor**：后端编译器，将计算图编译为优化后的 CUDA/CPU 内核
3. **算子融合**：将多个小操作合并为一个大的 CUDA kernel，减少内存访问和 kernel launch 开销

**编译模式：**

| 模式 | 优化程度 | 编译时间 | 适用场景 |
|------|----------|----------|----------|
| `default` | 适中 | 快 | 常规加速 |
| `reduce-overhead` | 高（减少 CPU 开销） | 中 | 小模型、CPU-bound |
| `max-autotune` | 最高（自动调优内核参数） | 很长 | 追求极致性能 |

**使用注意事项：**
- 首次运行有编译开销（可能几分钟），后续迭代才会加速
- 动态 shape（如动态 padding）可能触发频繁重编译，应使用 `dynamic=True` 或固定 shape
- 某些 Python 控制流（如数据相关的 if-else）可能导致 graph break，降低优化效果
- 不是所有模型都能被完美编译，复杂的自定义操作可能 fallback 到 eager 模式

典型加速效果：10-40%（取决于模型和硬件）

**MiniMind 中的实现：**

MiniMind 支持 `torch.compile` 加速训练：

```python
# 编译模型
model = MiniMindForCausalLM(config)
model = torch.compile(model)

# 之后正常训练即可，compile 对训练循环是透明的
for step, batch in enumerate(dataloader):
    with autocast_ctx:
        res = model(input_ids, labels=labels)
        loss = res.loss
    loss.backward()
    optimizer.step()
```

MiniMind 的模型架构（标准 Transformer + RMSNorm + SwiGLU）对 `torch.compile` 非常友好，因为：
- 没有复杂的动态控制流
- 使用固定长度截断（shape 固定）
- 标准操作符（Linear、Embedding、softmax 等）都有良好的编译支持

**追问方向：**
- `torch.compile` 和 ONNX Runtime、TensorRT 的区别？（compile 是训练 + 推理通用；ONNX/TRT 主要用于推理优化，压缩率更高但不支持训练）
- 如何判断 `torch.compile` 是否有效？（对比编译前后的 `tokens/sec`；使用 `TORCH_LOGS` 环境变量查看编译日志）

## Q30: 如何估算训练所需的 GPU 显存？给出 MiniMind 的计算过程

**标准答案：**

GPU 显存占用由四大部分组成（假设使用 Adam + 混合精度训练）：

**1. 模型参数**
- FP32 主权重：$4P$ 字节
- BF16/FP16 副本（autocast 临时生成）：$2P$ 字节

**2. 优化器状态（Adam）**
- FP32 一阶矩 m：$4P$ 字节
- FP32 二阶矩 v：$4P$ 字节
- FP32 参数副本：$4P$ 字节
- Adam 总计：$12P$ 字节

**3. 梯度**
- FP32 梯度：$4P$ 字节

**4. 激活值（中间计算结果）**
- 与 batch_size × seq_len × hidden_dim × num_layers 成正比
- 这是最大的变量，也是最容易 OOM 的来源
- 粗略估计：每个 Transformer 层每个 token 约需 $34 \times d_{model}$ 字节（含 attention 矩阵）

**MiniMind 64M 参数的显存估算：**

$P = 64 \times 10^6$

| 组件 | 计算 | 显存 |
|------|------|------|
| 模型参数 (FP32) | $4 \times 64M = 256 \text{MB}$ | 256 MB |
| 优化器状态 | $12 \times 64M = 768 \text{MB}$ | 768 MB |
| 梯度 | $4 \times 64M = 256 \text{MB}$ | 256 MB |
| 激活值（估算） | $bs \times seq \times hidden \times layers \times \text{factor}$ | ~1-2 GB |
| **总计** | | **~2.5-3.5 GB** |

具体到 MiniMind 预训练配置（`batch_size=32, max_seq_len=340, dim=768, n_layers=8`）：

激活值估算：
$$\text{activation} \approx 32 \times 340 \times 768 \times 8 \times 34 \approx 2.3 \text{GB}$$

但使用梯度累积时（`accumulation_steps=8`），实际每步的 `batch_size=32`（不是等效的 256），所以激活值按 32 计算。

加上 PyTorch 的 CUDA context（约 500MB-1GB）和碎片化开销：

$$\text{总显存} \approx 256 + 768 + 256 + 2300 + 700 \approx 4.3 \text{GB}$$

这远小于 3090 的 24GB，所以 MiniMind 可以轻松在单张 3090 上训练。

**常用经验公式（粗估）：**

$$\text{训练显存（GB）} \approx 20P \text{（字节）}$$

其中 $P$ 是参数量。MiniMind 64M → $20 \times 64M \approx 1.28 \text{GB}$（不含激活值）。

**MiniMind 中的实现：**

MiniMind 的模型配置和训练超参数直接影响显存用量：

```python
# 模型配置（影响参数量和激活值大小）
class MiniMindConfig:
    dim: int = 768          # 隐藏维度
    n_layers: int = 8       # Transformer 层数
    vocab_size: int = 6400  # 词表大小

# 训练配置（影响激活值大小）
# batch_size=32, max_seq_len=340 → 适配 24GB 显存
# accumulation_steps=8 → 等效 batch_size=256 而不增加显存
```

显存优化手段总结：

| 技术 | 节省对象 | MiniMind 是否使用 |
|------|----------|------------------|
| 混合精度 (AMP) | 激活值 | ✅ BF16 |
| 梯度累积 | 激活值 | ✅ steps=8 |
| 梯度检查点 | 激活值 | ❌（模型小，不需要） |
| ZeRO/FSDP | 参数+优化器 | ❌（单卡够用） |

**追问方向：**
- Gradient Checkpointing 的原理？为什么 MiniMind 不需要？（用时间换空间：前向传播不保存中间激活值，反向传播时重新计算。MiniMind 模型小，激活值不是瓶颈）
- 如何估算推理时的显存？（只需模型参数，无优化器和梯度。FP16 推理 MiniMind 64M ≈ 128MB + KV Cache）
